# 2. Project Data Context

The **Retail Intelligence Platform** is built on the **Olist Brazilian E-Commerce Public Dataset**, a multi-table transactional dataset representing marketplace orders, customers, products, payments, reviews, sellers, and geolocation data from a Brazilian e-commerce platform.

The raw dataset is relational in nature and consists of multiple linked entities. For the purpose of this project, these raw entities were analyzed and transformed into business-ready outputs used for:

- customer analytics
- repeat purchase and retention analysis
- RFM customer segmentation
- seller / marketplace performance analysis
- market basket analysis
- Streamlit dashboard reporting

# 6. Raw Source Table Definitions

---

# 6.1 `olist_customers_dataset.csv`

## Business Meaning
Stores customer identity and customer location details.

## Grain
One row per customer record.

## Important Notes
- A single **customer_unique_id** may appear across multiple orders through different **customer_id** values.
- Customer analysis in this project should generally be performed at the **customer_unique_id** level where possible.

## Key Columns

| Column | Type | Description | Notes |
| --- | --- | --- | --- |
| `customer_id` | string | Unique order-facing customer key | Links to orders table |
| `customer_unique_id` | string | Stable customer identity across purchases | Preferred customer analysis key |
| `customer_zip_code_prefix` | integer / string | Customer ZIP prefix | Used for regional analysis |
| `customer_city` | string | Customer city | Customer geography |
| `customer_state` | string | Customer state | Customer geography |

---

# 18. Business Rules and Assumptions

The following rules and assumptions are important for interpretation of the project outputs.

## 18.1 Customer Grain Rule
Customer analytics should prefer **customer_unique_id** when the goal is to analyze repeat behavior across multiple purchases.

## 18.2 Order Revenue Rule
Revenue calculations may originate from order item price, payment value, or aggregated spend logic depending on the notebook section. Users should align dashboard interpretation with the export logic used in the corresponding notebook.

## 18.3 Repeat Customer Rule
A repeat customer is generally defined as a customer with **more than one order**.

## 18.4 Product Category Translation Rule
Where available, English category names should be used for business-facing reporting instead of raw Portuguese category labels.

## 18.5 Basket Rule Interpretation
Association rules are directional and should be interpreted carefully:
- confidence reflects conditional likelihood
- lift reflects strength relative to random expectation
- high lift does not always imply high business volume

## 18.6 Seller Revenue Rule
Seller revenue in this project is typically derived from item-level sales associated with the seller.

# 19. Data Quality and Governance Notes

The following data quality considerations apply to the project.

## 19.1 Missing Values
Some raw Olist fields contain missing values, particularly in:
- product metadata
- review text fields
- delivery timestamps
- approval timestamps

## 19.2 Duplicate Risk
Some tables, such as payments and geolocation, can contain multiple rows for the same business entity and must be aggregated or handled carefully before reporting.

## 19.3 Column Name Normalization
Some dashboard pages include logic to standardize column names because notebook exports may vary slightly in naming conventions.

## 19.4 Export Dependency
The Streamlit application depends on the presence of report-ready CSV outputs. If a required report file is missing, the corresponding page may show a missing-file message or reduced functionality.

## 19.5 Notebook-to-Report Dependency
Business interpretation of output files should consider the notebook section that generated them. Output meaning can vary slightly depending on the exact export block used.
